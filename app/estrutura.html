Porque esta estrutura foi adotada

Separação de Responsabilidades

A aplicação web está separada da automação e da configuração do container, o que torna o projeto mais claro e mais fácil de manter.

Facilidade de Build

O Dockerfile está na raiz do projeto, permitindo que o build use toda a estrutura como contexto, incluindo a pasta app/.

Compatibilidade com GitHub Actions

O ficheiro do workflow está localizado em .github/workflows/, que é o local padrão reconhecido pelo GitHub Actions.

Documentação Centralizada

O README.md reúne a explicação geral do projeto, instruções de uso e informação sobre a pipeline.

Explicação da Estrutura

⚙️

.github/workflows

Contém o ficheiro YAML do GitHub Actions que define a pipeline CI/CD.

🖥️

app/

Contém todos os ficheiros da aplicação web: páginas HTML, CSS, JavaScript e imagens.

🐳

Dockerfile

Define como a imagem Docker é construída para servir a aplicação com Nginx.

📘

README.md

Documenta o projeto, a pipeline, a convenção de commits e a execução local.

Como os Ficheiros Participam no Processo

🌍

app/

É a interface visível do projeto. Tudo o que está nesta pasta será servido ao utilizador final através do Nginx.

🏗️

Dockerfile

Usa a imagem base do Nginx e copia a pasta app/ para dentro do container, criando a imagem Docker final.

🔁

ci-cd.yml

Automatiza a validação dos commits, o build da imagem e a sua publicação no GitHub Container Registry.

📦

GHCR

Recebe a imagem construída pela pipeline e guarda as versões publicadas do projeto.

Como os Componentes se Ligam

Aplicação

O conteúdo da pasta app/ representa a interface do projeto e é o que será servido ao utilizador final.

Docker

O Dockerfile copia o conteúdo da pasta app/ para dentro do container baseado em Nginx.

GitHub Actions

O workflow automatiza a validação dos commits e executa o build da imagem Docker.

GHCR

A imagem Docker construída pela pipeline é publicada no GitHub Container Registry.

Benefícios desta Organização

Esta estrutura torna o projeto claro, modular e fácil de apresentar. A aplicação fica separada da automação e da documentação, o que melhora manutenção, leitura e escalabilidade.

Além disso, permite que qualquer pessoa perceba rapidamente onde está o frontend, onde está o Dockerfile, onde está o workflow e como todo o processo CI/CD funciona.
