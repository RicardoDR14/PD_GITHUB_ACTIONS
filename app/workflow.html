Workflow GitHub Actions

Explicação do ficheiro YAML, da pipeline CI/CD e do papel de cada job e step.

Neste projeto, a pipeline garante que apenas commits válidos originam novas imagens Docker publicadas.

O que é o Workflow

O workflow é o ficheiro de configuração que define como a automação do projeto funciona no GitHub Actions. Ele determina quando a pipeline começa, que tarefas são executadas e em que ordem essas tarefas devem correr.

Neste projeto, o workflow é executado em cada push e divide-se em dois jobs: um para validar a mensagem do commit e outro para construir e publicar a imagem Docker.

CI e CD neste Projeto

🔍

Continuous Integration

A fase de CI está representada pela validação automática da mensagem de commit. Sempre que existe um push, o workflow verifica se o commit segue a convenção Conventional Commits antes de permitir qualquer build.

🚚

Continuous Delivery

A fase de CD está representada pela construção da imagem Docker e pela sua publicação automática no GitHub Container Registry, ficando pronta para ser utilizada.

Porque usar GitHub Actions

O GitHub Actions foi escolhido porque permite automatizar tarefas diretamente dentro do repositório, sem necessidade de ferramentas externas.

Permite definir triggers, jobs, steps e dependências, tornando a pipeline mais organizada, previsível e fácil de manter.

O que acontece se a validação falhar?

Se a mensagem do commit não respeitar o padrão definido, o job validate-commit termina com erro através de exit 1.

Como o job seguinte depende dele através de needs, o build e a publicação da imagem não chegam a acontecer.

Ficheiro YAML Utilizado

name: CI/CD Pipeline

on:
  push:
    branches:
      - "**"

jobs:
  validate-commit:
    runs-on: ubuntu-latest
    steps:
      - name: Check Conventional Commits format
        run: |
          COMMIT_MSG="${{ github.event.head_commit.message }}"
          if ! echo "$COMMIT_MSG" | grep -qE '^(feat|fix|docs|style|refactor|test|ci|chore)(\(.+\))?: .+'; then
            exit 1
          fi

  build-and-push:
    runs-on: ubuntu-latest
    needs: validate-commit
    steps:
      - uses: actions/checkout@v4
      - uses: docker/login-action@v3
      - uses: docker/metadata-action@v5
      - uses: docker/build-push-action@v5

Porque esta estrutura faz sentido?

Primeiro valida-se o commit (qualidade), depois constrói-se e publica-se (execução).

Isto garante controlo, eficiência e automatização do processo.
