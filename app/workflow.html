Explicação do ficheiro YAML, da pipeline CI/CD e do papel de cada job e step.

O que é o Workflow

O workflow é o ficheiro de configuração que define como a automação do projeto funciona no GitHub Actions. Ele determina quando a pipeline começa, que tarefas são executadas e em que ordem essas tarefas devem correr.

Neste projeto, o workflow é executado em cada push e divide-se em dois jobs: um para validar a mensagem do commit e outro para construir e publicar a imagem Docker.

Conceitos Essenciais

📄

Workflow

Ficheiro YAML que define toda a automação.

⚙️

Job

Conjunto de tarefas executado num runner.

✅

Step

Ação individual dentro de um job.

🖥️

Runner

Máquina virtual onde os jobs são executados.

Ficheiro YAML Utilizado

name: CI/CD Pipeline

on:
  push:
    branches:
      - "**"

jobs:
  validate-commit:
    name: Validate Commit Message
    runs-on: ubuntu-latest
    steps:
      - name: Check Conventional Commits format
        run: |
          COMMIT_MSG="${{ github.event.head_commit.message }}"
          echo "Commit message: $COMMIT_MSG"
          if ! echo "$COMMIT_MSG" | grep -qE '^(feat|fix|docs|style|refactor|test|ci|chore)(\(.+\))?: .+'; then
            echo "ERROR: Commit message does not follow Conventional Commits format."
            echo "Expected: <type>[optional scope]: <description>"
            echo "Valid types: feat, fix, docs, style, refactor, test, ci, chore"
            echo "Example: feat: add hello world page"
            exit 1
          fi
          echo "Commit message is valid."

  build-and-push:
    name: Build and Push Docker Image
    runs-on: ubuntu-latest
    needs: validate-commit
    permissions:
      contents: read
      packages: write
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract Docker metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=sha
            type=raw,value=latest,enable=${{ github.ref == 'refs/heads/main' }}

      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}

Explicação Step by Step

1. Trigger

O workflow começa com o evento push. Isso significa que qualquer alteração enviada para o repositório ativa automaticamente a pipeline.

2. Validate Commit

O primeiro job verifica se a mensagem do commit segue o padrão Conventional Commits. Se falhar, o workflow termina.

3. needs

O segundo job usa needs: validate-commit para garantir que só corre se a validação do commit passar com sucesso.

4. Checkout

O step actions/checkout descarrega o conteúdo do repositório para o runner.

5. Login no GHCR

O workflow autentica-se no GitHub Container Registry com o GITHUB_TOKEN, permitindo publicar a imagem.

6. Metadata

A action de metadata gera automaticamente as tags da imagem, incluindo a tag baseada no SHA e a tag latest na branch main.

7. Build and Push

No final, a imagem Docker é construída com base no projeto e publicada automaticamente no GHCR.

Porque esta estrutura faz sentido?

Esta organização permite separar responsabilidades dentro da pipeline. Primeiro, garante-se a qualidade e consistência do histórico de commits. Depois, apenas alterações válidas seguem para a etapa de build e publicação.

Além disso, o uso de permissões explícitas e da diretiva needs torna o workflow mais seguro, previsível e fácil de defender em contexto académico.
